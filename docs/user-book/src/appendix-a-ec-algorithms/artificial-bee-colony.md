# Artificial Bee Colony

<!-- source: crates/rlevo-evolution/src/algorithms/metaheuristic/abc.rs -->

Artificial Bee Colony (ABC) is a swarm metaheuristic for continuous optimisation
(Karaboga, 2005) modelled on the foraging behaviour of a honey-bee colony. Each
food source is a candidate solution; bees probe the neighbourhood of a source,
keep it if the neighbour is better, and abandon sources that stop improving.
Three roles divide the labour:

- **Employed bees** each tend one food source and probe its neighbourhood.
- **Onlooker bees** watch the employed bees and preferentially probe the
  *better* sources — this is where fitness biases the search.
- **Scout bees** abandon a source that has not improved for `limit` trials and
  replace it with a fresh random one — the colony's escape hatch from local
  optima.

Its key insight is shared with [Differential Evolution](differential-evolution.md):
the perturbation step is built from the *difference between two population
members*, so it self-calibrates to the current spread of the colony. ABC differs
in two respects — it perturbs **one coordinate at a time**, and it injects
diversity explicitly through the scout mechanism rather than relying on variation
operators alone.

## The neighbourhood step

Every candidate is generated by copying a target source `x_i` and perturbing a
**single, randomly chosen dimension** `j` toward (or away from) a randomly chosen
neighbour `x_k` (`k ≠ i`):

```math
v_{ij} = x_{ij} + \varphi \cdot (x_{ij} - x_{kj}), \qquad \varphi \sim U[-1, 1]
```

All other coordinates of `v_i` are copied unchanged from `x_i`. Two properties
fall out of this one line:

- **Self-scaling.** The magnitude `|x_{ij} − x_{kj}|` is large when the colony is
  spread out and shrinks as it converges, so the step size adapts to the search
  stage with no tuning — the same mechanism that makes DE step-size-free.
- **Reflection and contraction.** Because \\(\varphi\\) is drawn from
  \\([-1, 1]\\), the move can step *past* the neighbour (\\(\varphi < 0\\)),
  *toward* it, or land short — ABC both contracts and reflects, rather than only
  interpolating.

Perturbing a single coordinate makes ABC a fundamentally **coordinate-wise**
searcher: it is strong on *separable* landscapes (where coordinates can be
optimised independently) and slower on strongly rotated or non-separable
problems, where progress requires moving several coordinates together. This is
the central trade-off to keep in mind when choosing ABC over DE or CMA-ES.

## The two phases of a generation

`rlevo` fuses Karaboga's employed and onlooker phases into a single
`ask`/`tell` round that produces **`2 · pop_size` candidates** per generation:

| Phase | Candidates | Target selection | Perturbation |
|---|---|---|---|
| **Employed** | `pop_size` | every bee `i` is a target exactly once | `v_ij = x_ij + φ·(x_ij − x_kj)` |
| **Onlooker** | `pop_size` | tournament selection, fitness-biased | identical formula |

The only difference between the phases is *which* sources get probed. Employed
bees cover the colony uniformly; onlooker bees concentrate effort on the sources
that are already good, which is what turns ABC from a random walk into a directed
search. A consequence worth internalising: because onlooker targets are drawn by
tournament, the *same* source can be selected several times in one generation and
so receive several independent neighbourhood probes — the best improving one
wins (see *Greedy per-target replacement* below).

> **Roulette vs. tournament.** Canonical ABC selects onlooker targets by
> roulette (fitness-proportionate) probability. `rlevo` uses
> [`tournament_select`](differential-evolution.md) instead: it is the
> GPU-friendly, scale-invariant equivalent, reusing the same selection primitive
> as the rest of the engine and side-stepping roulette's sensitivity to fitness
> magnitude. Selection pressure is set by `tournament_size` rather than by the
> fitness spread.

## Configuration

```rust,no_run
use rlevo::evo::algorithms::metaheuristic::abc::AbcConfig;

// Explicit construction:
let config = AbcConfig {
    pop_size:        30,
    genome_dim:      10,
    bounds:          (-5.12, 5.12),
    limit:           150,           // (pop_size · genome_dim) / 2
    tournament_size: 3,
};

// Or use the canonical defaults:
let config = AbcConfig::default_for(30, 10);
```

| Field | Type | Typical range | Notes |
|---|---|---|---|
| `pop_size` | `usize` | 20 – 100 | Colony size; **≥ 2 required** (a neighbour `k ≠ i` must exist). Each generation costs `2 · pop_size` evaluations |
| `genome_dim` | `usize` | problem-defined | Dimensionality of the search space |
| `bounds` | `(f32, f32)` | problem-defined | Initial colony sampled here; candidates clamped here after perturbation |
| `limit` | `usize` | `pop_size · D / 2` | Scout trigger: a source with `trial > limit` is abandoned. Lower → more aggressive restarts |
| `tournament_size` | `usize` | 2 – 5 | Onlooker selection pressure; `1` would make onlookers uniform-random, defeating the fitness bias |

The scout `limit` is the parameter most worth tuning. Karaboga's canonical
`pop_size · D / 2` is a sensible default; lower it if the colony stagnates in
local optima, raise it if promising sources are being abandoned before they pay
off.

## Fitness convention

All strategies in `rlevo::evo` maximise a **canonical** fitness — higher is better. You declare a cost objective's direction with [`ObjectiveSense::Minimize`](https://docs.rs/rlevo-core) and the harness reconciles it at one chokepoint, so you never hand-negate. ABC's greedy acceptance test uses the same convention: a candidate replaces its target when its fitness is **no worse** (`cand ≥ current`), and the scout test fires when a source has gone `limit` generations without such an improvement. The best source is the argmax over the colony.

## Minimal example

```rust,no_run
use burn::backend::Flex;
use burn::tensor::{Tensor, TensorData, backend::Backend};
use rlevo::evo::algorithms::metaheuristic::abc::{AbcConfig, ArtificialBeeColony};
use rlevo::evo::fitness::BatchFitnessFn;
use rlevo::evo::strategy::EvolutionaryHarness;

type B = Flex;

/// Sphere function: f(x) = Σ xᵢ², minimum 0 at the origin.
struct SphereCost;

impl BatchFitnessFn<B, Tensor<B, 2>> for SphereCost {
    fn evaluate_batch(
        &mut self,
        population: &Tensor<B, 2>,
        device: &<B as burn::tensor::backend::BackendTypes>::Device,
    ) -> Tensor<B, 1> {
        let [pop_size, _] = population.dims();
        let data = population
            .clone()
            .into_data()
            .into_vec::<f32>()
            .unwrap_or_default();
        let dim = data.len() / pop_size;
        let fitness: Vec<f32> = (0..pop_size)
            .map(|row| (0..dim).map(|col| data[row * dim + col].powi(2)).sum::<f32>())
            .collect();
        Tensor::<B, 1>::from_data(TensorData::new(fitness, [pop_size]), device)
    }
}

fn main() {
    let device = Default::default();
    let config = AbcConfig::default_for(30, 10);   // limit = 150, tournament = 3

    let mut harness = EvolutionaryHarness::<B, _, _>::new(
        ArtificialBeeColony::<B>::new(),
        config,
        SphereCost,
        /* seed */ 13,
        device,
        /* max_generations */ 400,
    ).expect("valid config");

    harness.reset();
    while !harness.step(()).done {}

    let best = harness.latest_metrics().unwrap().best_fitness_ever;
    println!("best cost = {best:.4e}");   // converges below 1e-4
}
```

## Implementation notes

**One-dimension perturbation as a tensor mask.** Each candidate differs from its
target in exactly one coordinate, but the colony is perturbed as a single batched
tensor op. `build_candidates` constructs an `(n_cand, D)` boolean mask with a
`true` at `(row, dims[row])`, computes the full perturbed row
`base + φ·(base − neighbour)` for every coordinate, then uses `mask_where` to
splice in only the selected coordinate and keep the rest of `base`. The
per-candidate scalar \\(\varphi\\) is broadcast across `D` with
`unsqueeze_dim::<2>(1).expand(...)`. No host loop touches the genome data — the
whole `2 · pop_size × D` block is built in a handful of kernels.

**Greedy per-target replacement via gather.** `tell` does not rank the colony
globally. For each target source it keeps the single *best improving* candidate
(`cand ≥ current`, ties broken by strict `>` among competing candidates) and
leaves the source untouched otherwise. Applying the survivors avoids any
scatter-write: the implementation concatenates `[colony; candidates]` into one
`(pop_size + 2·pop_size, D)` tensor and builds an index vector `row_source[i]`
that is either `i` (keep) or `pop_size + cand_idx` (replace), then `select`s the
new colony in one gather. The same trick reapplies scout replacements from a
`[colony; fresh]` stack.

**Scout phase and `NEG_INF`-tagging.** A source whose `trial` counter exceeds `limit`
is overwritten with a fresh uniform sample, its counter reset to zero, and — the
subtle part — its cached fitness set to `f32::NEG_INFINITY`. Because the new source
has not been evaluated yet, tagging it with the worst possible sentinel guarantees
that *any* candidate probing it next generation registers as an improvement (higher
fitness), and excludes it from the best-so-far scan until it has a real score. The
next `ask` evaluates it for real.

**Two-cycle bootstrap.** ABC needs a fitness cache before its phases can run, so
the first cycle is a bootstrap: the first `ask` returns the initial colony
unchanged (it detects `state.fitness.is_empty()`), and the first `tell` uses the
received fitness to seed the cache and `best_genome` without running
employed/onlooker replacement. The employed and onlooker phases become active
from the **second `ask`** onward. [`EvolutionaryHarness`](../appendix-d-suppl/ask-tell-contract.md)
handles this transparently; a hand-written loop must run `ask → evaluate → tell`
through that bootstrap cycle before candidates appear.

**Reproducibility.** Every stochastic draw — initial colony
(`SeedPurpose::Init`), onlooker tournaments and neighbour/dimension/φ choices
(`SeedPurpose::Other`), and scout refills (`SeedPurpose::Replacement`) — comes
from an independent host-side sub-stream via `seed_stream`, never from
`Tensor::random` or `B::seed`. Two runs with the same root seed produce identical
trajectories regardless of how the population was evaluated in between.

## When to use

| Situation | Recommendation |
|---|---|
| Separable continuous landscape | ABC is a strong fit; coordinate-wise probing matches separable structure |
| Multi-modal with many local optima | ABC's scout restarts give good global exploration |
| Strongly non-separable / rotated problem | Prefer [DE](differential-evolution.md) (multi-coordinate differences) or CMA-ES (covariance) — single-coordinate moves are slow here |
| Fast local refinement near an optimum | ABC exploits weakly; consider the memetic wrapper or an ES `(μ+λ)` |
| Tight evaluation budget | Note ABC spends `2 · pop_size` evaluations per generation — roughly double a GA of the same colony size |
| Binary / combinatorial search | [Binary GA](binary-encoded-genetic-algorithm.md); ABC is defined over real-valued differences |
