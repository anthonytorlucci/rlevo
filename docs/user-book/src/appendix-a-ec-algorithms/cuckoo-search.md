# Cuckoo Search

<!-- source: crates/rlevo-evolution/src/algorithms/metaheuristic/cuckoo.rs -->

Cuckoo Search (CS; Yang & Deb, 2009) is a population metaheuristic built on a
brood-parasitism metaphor: each *nest* holds one candidate solution (an "egg"),
and every generation each nest lays a new egg some distance away by taking a
**Lévy flight** — a random step drawn from a heavy-tailed stable distribution.
Heavy tails mean most steps are small but occasional steps are very large, so the
search mixes fine local refinement with rare long-range jumps that can escape a
basin. After laying, a fraction of the worst nests are *abandoned* and
reinitialised, supplying fresh diversity.

The Lévy flight is the genuinely distinctive mechanism here; stripped of it, CS
is a biased random walk plus restart. It has **no gradient-biased update** — no
attraction toward a swarm best as in [PSO](particle-swarm-optimization.md), no
recombination — so it should not be expected to drive a smooth landscape to
machine precision within a modest budget. Treat it as a comparator and an
exploration-heavy sampler rather than a precision optimiser; for general
continuous work, [PSO](particle-swarm-optimization.md) is the recommended default
in `rlevo`.

`rlevo` implements it as `CuckooSearch` under
`rlevo::evo::algorithms::metaheuristic::cuckoo`, operating on real-valued nests
of shape `(pop_size, D)`.

## The Lévy flight step

`ask` proposes a new egg for every nest by adding a scaled Lévy-stable step to
its current position. The step is generated by **Mantegna's algorithm** (1994),
which produces an approximately Lévy-stable variate of index \\(\beta\\) from two
Gaussians:

```math
u \sim \mathcal{N}(0,\ \sigma_u^2), \qquad
v \sim \mathcal{N}(0,\ 1), \qquad
\text{step} = \frac{u}{\lvert v\rvert^{1/\beta}},
```

where the scale \\(\sigma_u\\) is the closed form that makes the ratio match a
Lévy distribution of index \\(\beta\\):

```math
\sigma_u = \left(
\frac{\Gamma(1+\beta)\,\sin\!\left(\pi\beta/2\right)}
     {\Gamma\!\left(\tfrac{1+\beta}{2}\right)\,\beta\,2^{(\beta-1)/2}}
\right)^{1/\beta}.
```

Each nest's position is then updated by a step of relative size \\(\alpha\\),
clamped back into the search bounds:

```math
\mathbf{x}_i' = \operatorname{clamp}\!\bigl(\mathbf{x}_i + \alpha\,\text{step}_i,\ \text{bounds}\bigr).
```

The Lévy index \\(\beta\\) (canonical \\(1.5\\), required in \\((0, 2)\\)) sets the
tail weight: smaller \\(\beta\\) means heavier tails and more frequent long jumps.
The step scale \\(\alpha\\) (default \\(0.05\\)) controls the overall move size and
is the main knob to tune to your domain — the paper's canonical \\(0.01\\) is
expressed relative to the search-space width.

## Greedy acceptance and nest abandonment

`tell` consumes the eggs' fitness and updates the nests in two stages.

**Greedy per-slot acceptance.** Each new egg competes only against the nest it
came from: egg \\(i\\) replaces nest \\(i\\) exactly when it is no worse,

```math
f(\text{egg}_i) \le f(\text{nest}_i).
```

This is a per-slot \\((1+1)\\) hill-climb, not a global tournament — a good egg in
one nest never displaces a different nest. The cached fitness is updated in step
with the accepted positions.

**Abandonment.** After acceptance, the \\(\lfloor p_a \cdot \texttt{pop\_size}
\rfloor\\) worst nests are abandoned and reinitialised with fresh uniform samples
from the bounds, where \\(p_a\\) is the abandonment probability (default
\\(0.25\\)). An abandoned slot's cached fitness is set to a sentinel \\(+\infty\\)
rather than being evaluated immediately. That sentinel is what reintegrates the
slot: on the next generation the greedy test \\(f(\text{egg}) \le +\infty\\) is
always true, so the Lévy step taken from the fresh position is unconditionally
accepted and the restarted nest rejoins the normal dynamics. Abandonment is the
only source of fresh diversity, since the Lévy step alone can only perturb
existing nests.

The best-so-far record is tracked separately over the **finite-fitness** slots
only (a sentinel \\(+\infty\\) slot is never mistaken for an incumbent) and is
re-pointed only on strict improvement.

## Configuration

```rust,no_run
use rlevo::evo::algorithms::metaheuristic::cuckoo::CuckooConfig;

// Explicit construction:
let config = CuckooConfig {
    pop_size:   30,
    genome_dim: 10,
    bounds:     (-5.12, 5.12),
    alpha:      0.05,   // step-size scale; tune to your domain
    beta:       1.5,    // Lévy index, must be in (0, 2)
    p_a:        0.25,   // abandonment fraction
};

// Or use the defaults (bounds = (-5.12, 5.12), α = 0.05, β = 1.5, p_a = 0.25):
let config = CuckooConfig::default_for(30, 10);
```

| Field | Type | Typical range | Notes |
|---|---|---|---|
| `pop_size` | `usize` | 15 – 50 | Number of nests; one evaluation per nest per generation |
| `genome_dim` | `usize` | problem-defined | Dimensionality of each nest |
| `bounds` | `(f32, f32)` | problem-defined | Initial nests sampled here; steps clamped here; abandoned nests refilled here |
| `alpha` (\\(\alpha\\)) | `f32` | 0.01 – 0.5 | Step-size scale; the primary knob. Larger → bolder moves |
| `beta` (\\(\beta\\)) | `f32` | (0, 2), ~1.5 | Lévy index; smaller → heavier tails, more long jumps |
| `p_a` (\\(p_a\\)) | `f32` | 0.1 – 0.5 | Fraction of worst nests abandoned each generation |

## Fitness convention

All strategies in `rlevo::evo` treat fitness as **cost** — lower is better.
Maximisation problems must be negated. The greedy acceptance keeps the lower-cost
egg and the best-so-far tracker is an argmin over finite slots.

## Minimal example

```rust,no_run
use burn::backend::Flex;
use burn::tensor::{Tensor, TensorData, backend::Backend};
use rlevo::evo::algorithms::metaheuristic::cuckoo::{CuckooConfig, CuckooSearch};
use rlevo::evo::fitness::BatchFitnessFn;
use rlevo::evo::strategy::EvolutionaryHarness;

type B = Flex;

/// Sphere function: f(x) = Σ xᵢ², minimum 0 at the origin.
struct SphereCost;

impl BatchFitnessFn<B, Tensor<B, 2>> for SphereCost {
    fn evaluate_batch(
        &mut self,
        population: &Tensor<B, 2>,
        device: &<B as burn::tensor::backend::BackendTypes>::Device,
    ) -> Tensor<B, 1> {
        let [pop_size, _] = population.dims();
        let data = population
            .clone()
            .into_data()
            .into_vec::<f32>()
            .unwrap_or_default();
        let dim = data.len() / pop_size;
        let fitness: Vec<f32> = (0..pop_size)
            .map(|row| (0..dim).map(|col| data[row * dim + col].powi(2)).sum::<f32>())
            .collect();
        Tensor::<B, 1>::from_data(TensorData::new(fitness, [pop_size]), device)
    }
}

fn main() {
    let device = Default::default();
    let mut config = CuckooConfig::default_for(30, 10);
    config.alpha = 0.2; // bolder steps help on a D=10 sphere

    let mut harness = EvolutionaryHarness::<B, _, _>::new(
        CuckooSearch::<B>::new(),
        config,
        SphereCost,
        /* seed */ 19,
        device,
        /* max_generations */ 800,
    );

    harness.reset();
    while !harness.step(()).done {}

    let best = harness.latest_metrics().unwrap().best_fitness_ever;
    // CS is a Lévy random walk + abandonment, not a precision optimiser:
    // expect a strong reduction from the random baseline, not machine zero.
    println!("best cost = {best:.4e}");
}
```

## Implementation notes

**Host-side Lévy sampling.** The two Gaussians are drawn on the host via
`seed_stream` and the step is uploaded with `Tensor::from_data`; the strategy
never calls `Tensor::random` or `B::seed` (the host-RNG convention). The
\\(\sigma_u\\) constant needs \\(\Gamma\\) at non-integer arguments, computed with a
Lanczos approximation accurate to \\(\sim 10^{-3}\\) over the valid \\(\beta\\)
range — more than enough, since \\(\sigma_u\\) only scales the step.

**Backend-parity caveat.** The fractional power \\(\lvert v\rvert^{1/\beta}\\) is
sensitive to FMA reordering: wgpu reductions can drift \\(\sim 10^{-3}\\) relative
from the `Flex`/ndarray path on the *same seed*. CS trajectories are therefore
not bit-identical across backends, and the backend-parity test relaxes its
tolerance for CS accordingly. Within a single backend, runs are fully
reproducible.

**Abandonment sentinel.** Reinitialised nests carry \\(+\infty\\) fitness rather
than being re-evaluated in the same `tell`. This keeps `tell` to one evaluation
batch per generation (the eggs) and lets the *next* generation's greedy step fold
the fresh position back in — see [above](#greedy-acceptance-and-nest-abandonment).

**Two-cycle bootstrap.** Like the other swarm strategies, CS needs a fitness
cache before it can move. The first `ask` returns the initial nests unchanged
(it detects an empty fitness cache) and the first `tell` seeds the cache and the
best-so-far record without running acceptance or abandonment. Lévy proposals go
live from the **second `ask`**.
[`EvolutionaryHarness`](../appendix-d-suppl/ask-tell-contract.md) handles this
transparently; a hand-written loop must run the bootstrap cycle first.

**CubeCL kernel is deferred.** A fused `levy_flight_cube` Mantegna sampler is
designed under the metaheuristic `kernels/` directory (gated behind the
`custom-kernels` feature) but not yet implemented; CS uses the host-sampling path
above until it lands.

**Reproducibility.** Initial nests (`SeedPurpose::Init`), the per-generation Lévy
draws (`SeedPurpose::Mutation`, keyed by the generation counter), and the
abandonment refill (`SeedPurpose::Replacement`, also generation-keyed) come from
independent host sub-streams. Two runs with the same seed and config produce
identical trajectories on a fixed backend.

## When to use

| Situation | Recommendation |
|---|---|
| General continuous optimisation | Prefer [PSO](particle-swarm-optimization.md) — fewer caveats, a real attractor, faster convergence |
| Want heavy-tailed exploration / rare long jumps | Cuckoo Search — the Lévy flight is its reason to exist; raise \\(\alpha\\) or lower \\(\beta\\) for bolder mixing |
| Rugged multimodal landscape, exploration over precision | CS's Lévy jumps plus abandonment resist sticking in one basin; pair with a local refiner if you need precision |
| Need high-precision convergence on a smooth landscape | [CMA-ES](cma-es.md) or [DE](differential-evolution.md) — CS has no gradient-biased update and plateaus early |
| Strong variable dependencies | [CMA-ES](cma-es.md) — CS perturbs each coordinate independently |
| Reproducible results across wgpu and ndarray backends | Be aware CS is **not** bit-identical across backends (the fractional-power caveat); use a single backend if exact parity matters |
| Binary / combinatorial search | [Binary GA](binary-encoded-genetic-algorithm.md) — CS is defined over continuous nests |
